# Status Sync — 週次進捗集約

毎週金曜17時、全Lite PJの状況を `#lite-lapidaristデリバリー-cc` に1投稿で集約する。

## 入力

- 各PJのSlackチャンネル（顧客側 or 社内スレッド）の直近1週間の発言
- Zoom録画 / 文字起こし（直近1週間の定例分）
- カレンダーの実施済み定例MTG一覧
- ダッシュボードの Lite一覧（`ai-dashboard.html`）

## アウトプット

```
# Weekly Lite Status — YYYY/MM/DD

## サマリ
- 順調: X件 / 要注意: X件 / 赤信号: X件

## PJ別

### [PJ名 1] — [顧客名]
- 進捗: XX% / マイルストーン: [今週完了 X件 / 持ち越し X件]
- ハイライト: [1-2行]
- ブロッカー: [なし / 〜のため停滞]
- 次週フォーカス: [一行]

### [PJ名 2] — [顧客名]
（同様）

...

## 全体トピック
- [社内決定が必要な事項 / 参照: ../_shared/stakeholder-map.md]
- [リスクログ更新: ../project-mgmt/risk-log.md]
```

## AIプロンプト例

### 全PJ集約

```
以下の入力から、上記フォーマットで Weekly Lite Status を生成して:

- PJ一覧: [ダッシュボードLite一覧 リンク]
- 各PJのSlackログ（直近7日）: [リンク x N]
- 定例議事録（直近7日）: [リンク x N]

各PJの「進捗」は、キックオフ時に定義したマイルストーン進捗 [リンク] と照合して算出。
「ブロッカー」は、Slack発言から「困っている / 待っている / 止まっている」のキーワードを抽出。
「順調 / 要注意 / 赤信号」は以下基準:
- 順調: マイルストーン遅延なし、ブロッカーなし
- 要注意: マイルストーン遅延 ≤ 3日、または軽微なブロッカーあり
- 赤信号: マイルストーン遅延 > 3日、または重大ブロッカーあり
```

### Zoom議事録要約

```
Zoom録画の文字起こし [リンク] から、
- Decision（決定事項）
- Action（持ち帰りタスク 担当・期日付き）
- Risk（議論で出た懸念）
の3区分で要約。各項目1〜3行で。
顧客側にも共有可能なフォーマット（伏字なし・敬語）。
```

### 個別PJ深掘り

```
[PJ名] について、過去4週の議事録 [リンク x4] を時系列で読み、
- 繰り返し出ているトピック（3つ）
- 解決された課題 vs 未解決の課題
- 顧客側のトーンの変化（前向き / 中立 / 不安）
を分析して。次の定例で扱うべき1つを推奨。
```

## 留意点

- 集約は **金曜17時** に固定（みんなの「読む準備」のリズムを作る）
- 1投稿に収まらない場合はスレッドで分割。チャンネルを汚さない
- **赤信号PJ** は、Status Sync投稿後30分以内に Delivery Lead が `@here` で次アクション提案を投稿
- 「順調」が3週続いたら、それは本当に順調か疑う（盲点が育っている可能性）

## 入力が揃わない場合

- Slackログが空 = PJが死にかけている可能性。即 Delivery Lead にDM
- Zoom録画がない週 = 定例MTGがスキップされた。理由をPJ担当に確認
- マイルストーンが定義されていないPJ = キックオフが不完全。`../delivery/kickoff.md` のチェックリストを再実行
