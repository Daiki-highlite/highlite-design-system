# Risk Log — リスク台帳

PJ・経営・組織のリスクを **早期に書き留め、判断のタイミングを共有** する。解決を急がない。気づくことが価値。

## リスクの定義（Highlite版）

「**まだ起きていないが、放置すると顧客・メンバー・事業に損失をもたらす可能性のある事象**」

含むもの:
- スケジュール遅延の予兆
- スコープ外要求の蓄積
- メンバー稼働の偏り
- 顧客側の意思決定停滞
- 入金遅延の兆候
- パートナーの稼働離脱予兆
- 事業上の意思決定の先送り

含まないもの:
- 既に起きた問題（→ Issue として別管理）
- 抽象的な不安（→ 具体化されない限りリスクではない）

## 台帳の構造

PJ別 + 全社共通の2層管理。Notion or Markdown表で運用。

```
| ID | 種別 | 内容 | 影響度 (H/M/L) | 発生確率 (H/M/L) | スコア | 起票者 | 起票日 | 監視担当 | 対応方針 | 状態 |
|---|---|---|---|---|---|---|---|---|---|---|
```

スコア = 影響度 × 発生確率（H=3, M=2, L=1。最大9）

| スコア | 取り扱い |
|---|---|
| 9 | 即時 `@here` + 翌営業日までに対応方針確定 |
| 6-8 | 週次Status Syncで明示、月次レビューで判断 |
| 3-5 | 月次レビューでまとめて確認 |
| 1-2 | 定期監視のみ |

## AIプロンプト例

### 新規リスク起票

```
以下の状況から、リスクログ起票案を作って:

- 状況: [PJ / 顧客 / メンバー / 事業 のどこで何が起きそうか]
- 兆候: [観察された具体的な事実]
- 想定される最悪のケース: [起きると何が壊れるか]

出力フォーマット:
- 種別: [PJ進捗 / スコープ / 稼働 / 顧客関係 / 入金 / パートナー / 経営判断]
- 内容: [1-2文で]
- 影響度評価: [H/M/L + 理由]
- 発生確率評価: [H/M/L + 理由]
- 推奨される監視担当: [../_shared/stakeholder-map.md と照合]
- 対応方針案: [予防 / 軽減 / 受容 / 転嫁 のどれか + 一文]
```

### 週次レビュー

```
リスクログ [リンク] の全エントリを読み、
- 状態が「監視中」のまま2週間以上経過しているもの
- スコアが上昇傾向のもの
- 互いに連動するリスク（同じ顧客 / 同じメンバー / 同じ時期）
を抽出し、Status Sync投稿に含めるサマリを生成して。
```

### 月次レビュー

```
過去1ヶ月のリスクログ変動から、
- 新規起票されたリスク（種別別カウント + 主なもの3つ）
- クローズされたリスク（解消理由のパターン）
- 構造的リスク（同じ種別が繰り返し起票されている → 仕組みの問題）
- 全社で次月優先して対処すべき3つ
をレポート化して。経営判断会議のアジェンダ用。
```

## 注意

- **リスクは「悪いこと」ではなく「準備するための情報」**。起票しても評価されない / されたら困る空気を絶対に作らない
- **誰でも起票できる** — パートナー含む全員が対象
- **匿名起票は許容しない** — 起票者が分からないと、深掘りができない
- **クローズ理由は必ず書く** — 「解消した」だけでなく、「なぜ / どう」を残すと組織知になる

## アンチパターン

- リスクログが「将来やりたいこと」リストになる → リスクの定義に戻る
- 同じリスクが何度も起票される → 仕組みの問題。月次レビューで「ルール変更」を検討
- 高スコアリスクを「いつか対処する」で寝かせる → スコア9になった時点で判断必須
