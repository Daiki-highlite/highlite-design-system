# Proposal Draft — Ore 04

BANTC合格案件に対して、提案書ドラフト〜送付〜フォローアップまでを設計する。

## 提案書スケルトン（標準型）

```
1. 課題認識        ← Disco Noteから引用、相手の言葉で
2. 解決の方針      ← 抽象→具体→抽象 の構成
3. 提供価値        ← 定性 + できれば定量
4. 投資            ← 金額 + 内訳 + ROI試算
5. 体制とスケジュール ← 主担当 / マイルストーン
6. 次の一歩        ← 具体的に「いつまでに何をすれば動き出せるか」
```

## AIプロンプト例

### スケルトン生成

```
Disco Note [リンク] と類似事例3件 [リンク x3] を踏まえ、
上記6章構成の提案書スケルトンをMarkdownで作って。
- 各章の見出し直下に「狙い（30字以内）」を1行で
- 第3章「提供価値」は定性3つ + 定量1つを必ず含める
- 第4章「投資」は3案併記（最小 / 標準 / 充実）
```

### トーンレビュー

```
提案書ドラフト [リンク] を ../_shared/brand-voice.md 基準でレビューし、
トーンが外れている箇所を以下の3列で指摘:
| 現状 | 修正案 | 理由 |

特に：
- 過度敬語 / 装飾過多
- 「効率化」「最適化」など glossary.md の禁則語
- 「絶対」「100%」など過度な約束
```

### 見積根拠

```
見積金額 [数値] の根拠を、以下3観点で1ページにまとめて:
- 工数内訳（フェーズ別 / 人月）
- 類似案件比較（過去事例 [リンク] との対比）
- ROI試算（顧客側の期待効果 / ペイバック期間）

ブランドトーンに沿い、過度な売り込みにならないこと。
```

### クライアント送付メール

```
提案書 [リンク] を送付するメールを起案して。
- 宛先: [氏名・役職]
- Disco Note [リンク] からの主要引用を1〜2文入れる（「先日いただいた『〜』というお言葉から…」）
- 添付物・別添資料の説明を箇条書きで
- 次のステップ（質疑回答方法 / 検討期限の確認）を明示
- 結びは「ご一緒できることを楽しみにしています」程度の温度感
```

## 社内レビューフロー（小規模組織版）

1. **ドラフト完成** → `#ore-リード・アサイン情報-cc` にレビュー依頼（48hデッドライン）
2. **レビュアー**: DH（必須）+ Delivery Lead（推奨）
3. **指摘反映** → ファイナル化
4. **送付前最終チェック**: 金額・期日・体制図の固有名詞を再確認
5. **送付後** → 同チャンネルに「[企業名] 提案送付（金額・確度）」報告

## フォローアップ

- 送付 **+3日**: 開封確認・読了確認の軽いメッセージ（任意）
- 送付 **+7日返答なし**: 担当者DMにフォローアップタスク発行
- 送付 **+14日返答なし**: クライアントへ「検討状況いかがですか」を送る + Disco Note読み返し
- 送付 **+30日返答なし**: Crysta索敵対象に切り替え

## 注意

- **提案書は資料ではなく、対話の続き**である。Disco Note の温度感を保つ
- **金額の出し方**: 範囲提示 → 推奨案、の順。最初から1点提示しない
- **競合がいる場合**: 比較で勝とうとせず、Highlite独自の「磨き方」を語る
- 提案書送付後、契約締結までの期間は **既存PJメンバーをこの案件にアサインしない**（受注後に動く前提で計画する）
