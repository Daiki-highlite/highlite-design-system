# Case Study — 事例記事化

Crysta 01「実績開示の許可」を経た案件を、Web掲載・営業資料・SNSで使える形に**結晶化**する。

## 前提条件

- 公開許諾書を取得済み（実名 / 業種匿名 / 数値レンジのいずれかで合意）
- 開示範囲が明文化されている
- 顧客側の最終確認を得るプロセスを組み込む

## 標準構成（800字版）

```
1. 課題（120字）
   - 顧客が抱えていた具体的な状態
   - その状態が続くと何が困るか

2. アプローチ（240字）
   - Highliteとして何を「磨いた」か
   - 選択した道と、選ばなかった道
   - 重要だった意思決定1つ

3. 成果（240字）
   - 定性的変化（顧客の言葉で）
   - 定量的変化（数値レンジで可）
   - 副次的な発見

4. 顧客の声（160字 / 引用）
   - 顧客側担当者の言葉そのまま
   - 形容詞を盛らず、具体的な情景

5. 余韻（40字）
   - その後の関係性 / 次に向かっていること
```

## AIプロンプト例

### ドラフト起案

```
合意済み開示条件 [本文] と PJ概要 [リンク] と Disco Note [リンク] と
最終納品物 [リンク] から、上記5章構成の事例記事ドラフト（800字）を起案。

「アプローチ」では、選んだ道だけでなく**選ばなかった道**を必ず1つ書く。
「顧客の声」は、議事録や検収後コメント [リンク] から実際の発言を引用。
形容詞「素晴らしい」「最高」は使わない。具体的な情景で語る。
ブランドトーン: ../_shared/brand-voice.md
```

### SNSへの分解

```
事例記事 [リンク] から、SNS用の引用ポスト案を3パターン:

- LinkedIn（300字 / 業界向け）: アプローチの意思決定にフォーカス
- X（140字 + リンク）: 顧客の声から1文 + 余韻
- noteの冒頭3段落（500字）: 課題からの導入を物語仕立てで

各案、Highliteを主語にしすぎない。「[顧客]が...」で始めるパターンを優先。
```

### 営業資料への組み込み

```
事例記事 [リンク] を、営業資料スライド3枚分に再構成:

- 1枚目: 課題ビフォー / 顧客の状態を1行 + 当時の言葉を引用
- 2枚目: アプローチ / プロセス図 or キービジュアル + 重要意思決定1つ
- 3枚目: 成果アフター / 定量変化 + 顧客の声

文字は最小限、視覚で伝わる構成にして。
```

### バリエーション展開

```
事例記事 [リンク] から、ターゲット別バリエーションを起案:

- 経営層向け（投資対効果を強調 / 800字）
- 実務担当者向け（プロセス詳細を強調 / 1200字）
- デザイナー向け（クリエイティブの意思決定を強調 / 1000字）

骨格は共通、強調点と語彙を変えるだけ。
```

## 制作プロセス

```
1. 公開許諾書取得（Crysta 01）
2. ドラフト起案（AI支援）
3. 社内レビュー（Delivery Lead + DH）
4. 顧客に最終確認依頼（修正の余地を明示）
5. 修正反映 → ファイナル
6. 公開（Web / SNS / 営業資料）
7. 公開後の反響をCrystaレコードに紐付け
```

## 注意

- **顧客の言葉を**改変しない** — 編集は語尾の整え程度に留める
- **Highliteを主役にしすぎない** — 顧客の物語の中で、Highliteは触媒
- **数値は丁寧に** — 単位・期間・比較対象を必ず明記
- **失敗や葛藤も書ける範囲で書く** — 完璧な物語は嘘っぽい
- **公開前に必ず顧客側の最終確認** — 「事前に共有した内容と表現が変わっていないか」のチェックを依頼
